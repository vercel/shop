The template includes opt-in customer authentication using the framework-neutral session and OAuth helpers from `@shopify/hydrogen/customer-account`. Customers can sign in through Shopify, view their profile, order history, and address book, refresh expired access tokens, and sign out of both the storefront and Shopify identity provider.

Authentication is disabled by default. The canonical default lives in `shop.config.ts`, and a deployment can override it with `NEXT_PUBLIC_ENABLE_AUTH`. When disabled, the storefront works as a guest-only experience and no auth UI is rendered.

## Configuration

Set the feature flag, Customer Account API client ID, session secret, and public storefront URL:

```bash
NEXT_PUBLIC_ENABLE_AUTH="1"
CUSTOMER_ACCOUNT_SESSION_SECRET="[redacted]"
SHOPIFY_CUSTOMER_ACCOUNT_API_CLIENT_ID="your-customer-account-client-id"
NEXT_PUBLIC_BASE_URL="https://your-domain.com"
```

Shopify provides `SHOPIFY_CUSTOMER_ACCOUNT_API_CLIENT_ID`, but not `CUSTOMER_ACCOUNT_SESSION_SECRET`. Generate that private, app-owned secret with `openssl rand -base64 32` — the template uses it to derive the AES-256-GCM key that encrypts Hydrogen's OAuth session data in HttpOnly cookies. Keep it server-only and consistent across deployment instances; rotating it invalidates every existing customer session. `next.config.ts` throws at build time whenever auth is enabled without the session secret and client ID.

Customer Account OAuth requires a public HTTPS origin. Use a tunnel for local OAuth testing and set `NEXT_PUBLIC_BASE_URL` to that registered origin.

To enable it in Shopify Admin:

1. Enable customer accounts: **Settings → Customer accounts → Edit**, choose **Customer accounts**, and Save
2. Install the **Headless** sales channel from the Shopify App Store
3. Go to **Sales channels → Headless → (your storefront) → Customer Account API**
4. Use a public Customer Account API client; Hydrogen performs authorization code + PKCE without a client secret
5. Set the callback URI to `{YOUR_DOMAIN}/account/authorize` and the logout URI to `{YOUR_DOMAIN}/`
6. Copy the client ID to `SHOPIFY_CUSTOMER_ACCOUNT_API_CLIENT_ID`, and confirm the store domain matches `SHOPIFY_STORE_DOMAIN`

Shopify does not support wildcard callback or logout URIs — register each production, preview, or tunnel origin that needs authentication. See [Environment Variables](/docs/reference/env-vars) for the full variable reference.

## How it works

**No client-side auth SDK.** Sign-in is a normal GET navigation to `/account/login`; sign-out is a same-origin POST form to `/account/logout`. The universal `isAuthEnabled` flag (`lib/auth/index.ts`) re-exports `shopConfig.auth.enabled`, safe to import from server and client code alike. When it's false: the account icon disappears from the nav, the Hydrogen auth routes and protected account pages return `404`, and the shared session/access-token helpers in `lib/auth/server.ts` fail before initializing Hydrogen — no auth-related code runs at request time.

**The session lives in an encrypted cookie, not a server store.** `/account/login` asks Hydrogen to generate state, nonce, and PKCE values, which the app encrypts into chunked AES-256-GCM HttpOnly cookies. Shopify redirects to `/account/authorize`, where Hydrogen validates state, nonce, issuer, audience, and ID-token expiry before exchanging the authorization code — access, refresh, and ID tokens then replace the pending-login state in that same cookie. Server Components only perform read-only login and token checks; they never mutate cookies.

**Token refresh happens out-of-band, not inline.** `isCustomerLoggedIn()` treats a refreshable session as logged in, so the nav doesn't flicker to signed-out when the access token expires. `requireCustomerAccessToken()` redirects through `/account/refresh` when only a refresh token remains, where Hydrogen rotates tokens and commits the updated cookie before returning to the account page. Logout mirrors this: the sign-out button POSTs to `/account/logout`, which clears the local session and redirects through Shopify's logout endpoint with `id_token_hint`, ending both the storefront session and the Shopify SSO session.

```ts
import {
  isCustomerLoggedIn,
  requireCustomerAccessToken,
  requireCustomerSession,
} from "@/lib/auth/server";

const loggedIn = await isCustomerLoggedIn();
await requireCustomerSession();
const accessToken = await requireCustomerAccessToken("/account/orders");
```

Use `isCustomerLoggedIn()` for read-only UI state, `requireCustomerSession()` for account-page gates, and `requireCustomerAccessToken()` immediately before private Customer Account API work.

> **Guardrails.** Never expose access, refresh, or ID tokens to Client Components or browser-readable storage. Keep Server Components on the read-only session-manager path — token refresh belongs in `/account/refresh`. Keep login as GET and logout as same-origin POST, and keep the callback/logout URIs registered exactly in Shopify. Rotate `CUSTOMER_ACCOUNT_SESSION_SECRET` deliberately, since rotation invalidates every existing session, and keep `NEXT_PUBLIC_ENABLE_AUTH` public so server/client feature gates agree under cache components.

## Out of the box

- **Auth routes** — `/account/login` (GET, starts PKCE and redirects to Shopify), `/account/authorize` (GET, validates the OAuth callback), `/account/refresh` (GET, rotates an expired access token), and `/account/logout` (POST, clears the session). All four are installed via Hydrogen's standard handlers through `handleShopifyRoutes` in `app/account/[action]/route.ts`.
- **Account pages** — `/account` (redirects to profile), `/account/profile` (edit name, read-only email), `/account/orders` (paginated order history), `/account/orders/[id]` (order detail), and `/account/addresses` (address book with create, edit, delete, and default selection). The `(authenticated)` route group gates these without claiming the auth-handler paths.
- **Customer Account API data** — a separate GraphQL endpoint and schema from the Storefront API. `lib/shopify/customer-account.ts` wraps Hydrogen's `createCustomerAccountClient`; `lib/shopify/operations/customer.ts` queries and mutates profile, orders, and addresses, resolving a usable token through `requireCustomerAccessToken()` on every call; `lib/shopify/transforms/customer.ts` maps responses to domain types. Order and profile pages are Server Components in Suspense; address and profile editing use client forms backed by server actions in `lib/customer/action.ts` that validate input, surface Shopify `userErrors`, and revalidate affected paths. Customer responses are personalized and never publicly cached.

## Common customizations

- **Additional account data** — store credit, subscriptions, draft orders, or richer order fields. Add an operation in `lib/shopify/operations/customer.ts`, a transform in `lib/shopify/transforms/customer.ts`, and a matching domain type in `lib/types.ts`, validated against the live Customer Account API schema. Route mutations through a server action in `lib/customer/action.ts` so validation and revalidation stay centralized.
