All commerce data flows through a `CommerceProvider` interface defined in `lib/commerce/`. Each provider implements this interface, keeping the storefront decoupled from any specific backend. Operations are called through the commerce abstraction, and providers handle the underlying API communication.

## Commerce operations

```ts
import { commerce } from "@/lib/commerce";

const product = await commerce.getProduct(handle, locale);
```

The commerce object exposes typed operations for all storefront needs: products, collections, cart, search, and menus. Each provider implements these operations against its own API.

## Provider architecture

Providers live in `lib/commerce/providers/<name>/` and implement the `CommerceProvider` interface. A typical provider includes:

- **Client** â€” the API fetch wrapper with authentication and error handling
- **Operations** â€” functions that call the provider's API and return data
- **Transforms** â€” converters from provider-specific response shapes to domain types

For example, a GraphQL-based provider would send `POST` requests to its API endpoint with the appropriate authentication headers.

## Writing an operation

Operation functions follow a consistent pattern regardless of provider:

```ts
import { cacheLife, cacheTag } from "next/cache";
import { defaultLocale, getCountryCode, getLanguageCode } from "@/lib/i18n";

export async function getProduct(handle: string, locale: string = defaultLocale) {
  "use cache: remote";
  cacheLife("max");
  cacheTag("products", `product-${handle}`);

  const data = await providerFetch({
    operation: "getProductByHandle",
    variables: {
      handle,
      country: getCountryCode(locale),
      language: getLanguageCode(locale),
    },
  });

  return transformProductDetails(data.productByHandle);
}
```

Key elements:

1. **`"use cache: remote"`** â€” opts the function into Next.js request-level caching
2. **`cacheLife("max")`** â€” caches indefinitely until manually revalidated
3. **`cacheTag(...)`** â€” assigns tags for granular invalidation via `revalidateTag()`
4. **Transform** â€” convert the provider response to a domain type before returning. Components never import provider-specific types directly.

## Caching

All read operations use `"use cache: remote"` with `cacheLife("max")` and one or more cache tags. Tags follow a hierarchy:

| Tag pattern | Scope |
|-------------|-------|
| `products` | All product queries |
| `product-{handle}` | Single product by handle |
| `product-{numericId}` | Single product by numeric ID |
| `recommendations-{handle}` | Product recommendations |
| `collections` | All collection queries |
| `collection-{handle}` | Single collection |
| `menus` | Navigation menus |
| `cart` | All cart operations |

Invalidate with `revalidateTag("product-blue-tee")` or `updateTag("cart")` in server actions. Cart mutations must always call `invalidateCartCache()` from `lib/cart-cache` -- this is a hard requirement.

## Mutations

Cart mutations use the same provider fetch function but are not cached. They call the provider's cart line add, update, and remove operations:

```ts
export async function addToCart(lines: CartLineInput[], cartId: string, locale: string) {
  const data = await providerFetch({
    operation: "addToCart",
    variables: { cartId, lines, country: getCountryCode(locale) },
  });

  invalidateCartCache();
  return transformCart(data.cart);
}
```

The provider returns the full updated cart in every mutation response, so there's no need for a follow-up query.

## Error handling

The client handles errors at two levels:

**HTTP errors** â€” a non-200 response throws immediately with the status code and text.

**API errors** â€” if the response contains errors and no data, the client throws. If there are errors but data is also present (partial success), it logs a warning and returns the partial data.

Individual operations add their own handling on top. For example, `getCart()` wraps the call in try-catch and returns `undefined` on failure, while `getProduct()` throws if the product isn't found.

## Transforms

Every operation converts the provider response to a domain type before returning. Transforms live in `lib/commerce/providers/<name>/transforms/`:

| Transform | Input -> Output |
|-----------|---------------|
| `transformProductDetails` | Provider product -> `ProductDetails` |
| `transformCart` | Provider cart -> `Cart` |

Domain types are defined in `lib/types.ts` and are provider-agnostic. Components import these types, never the provider-specific response shapes. This keeps the UI decoupled from the API layer.

## Key files

| File | Purpose |
|------|---------|
| `lib/commerce/index.ts` | Commerce provider interface and exports |
| `lib/commerce/providers/<name>/client.ts` | Provider-specific fetch wrapper with auth and error handling |
| `lib/commerce/providers/<name>/operations/products.ts` | Product queries: single, batch, search, recommendations |
| `lib/commerce/providers/<name>/operations/cart.ts` | Cart queries and mutations |
| `lib/commerce/providers/<name>/operations/collections.ts` | Collection queries |
| `lib/commerce/providers/<name>/operations/search.ts` | Predictive search |
| `lib/commerce/providers/<name>/operations/menu.ts` | Navigation menu queries |
| `lib/commerce/providers/<name>/transforms/product.ts` | Provider product -> domain type |
| `lib/commerce/providers/<name>/transforms/cart.ts` | Provider cart -> domain type |
| `lib/cart-cache.ts` | `invalidateCartCache()` for mutation cache busting |
