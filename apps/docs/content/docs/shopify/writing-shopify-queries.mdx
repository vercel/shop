This guide walks through adding a new Shopify Storefront API operation from scratch. We'll fetch a shop's privacy policy as a concrete example.

If you're using a coding agent, run `/vercel-shop:shopify-graphql-reference` in Claude Code or ask the agent to use the `shopify-graphql-reference` skill first. The skill is the reference source for Shopify GraphQL work and uses `shopify-ai-toolkit` for live schema checks while keeping the agent anchored to the template guardrails.

## 1. Check the schema

Use `shopify-ai-toolkit`, Shopify's GraphQL docs, or your preferred GraphQL explorer to confirm the field exists before writing the query. For our example, the `QueryRoot` type has a `shop` field with a `privacyPolicy` property that returns a `ShopPolicy`:

```graphql
type ShopPolicy {
  body: String!
  handle: String!
  title: String!
  url: URL!
}
```

Never guess field names. The live Shopify schema is the source of truth.

## 2. Write the query

Create the GraphQL query as a template string. Two conventions make it work with codegen (step 4):

- Start the string with the `#graphql` comment tag so codegen picks it up.
- End the declaration with `as const` so its literal type flows through fragment interpolation.

Interpolate any shared fragments from `lib/shopify/fragments.ts` at the top. Include `@inContext` if the data is locale-sensitive:

```ts
const PRIVACY_POLICY_QUERY = `#graphql
  query getPrivacyPolicy($country: CountryCode, $language: LanguageCode)
    @inContext(country: $country, language: $language) {
    shop {
      privacyPolicy {
        title
        body
        handle
      }
    }
  }
` as const;
```

Queries must be **named** (`getPrivacyPolicy`) — codegen skips anonymous operations.

Keep the document **static**. Codegen can only parse a query it can read at rest, so don't assemble one with runtime string interpolation (a ternary that swaps in a field, a computed selection set built from config). Pass the dynamic part as a GraphQL variable, or split it into two static `#graphql` documents and choose one at the call site.

## 3. Define the response type

Type the shape of `data` that the GraphQL response will return:

```ts
interface ShopifyPrivacyPolicyResponse {
  shop: {
    privacyPolicy: {
      title: string;
      body: string;
      handle: string;
    } | null;
  };
}
```

This is a local type scoped to your operation file - it maps directly to the GraphQL response shape. You pass it to `storefront.request<T>()` (step 5) as the explicit response type.

## 4. Validate the query with codegen

Run codegen to check your query against the live Storefront schema:

```bash
pnpm --filter template codegen
```

Codegen parses every `#graphql`-tagged query and **fails the command if a field, argument, or type doesn't exist** — this is what enforces "never guess field names" automatically. It writes generated types to `lib/shopify/types/generated/`, which is **gitignored**: nothing in the app imports it, so it's purely the validation output. Codegen also runs automatically at dev and build time — `pnpm dev` regenerates on start (non-fatal if you're offline) and `pnpm build` runs it as a **hard gate** that fails on schema drift — so you rarely run it by hand. Use `pnpm --filter template codegen:watch` while iterating.

## 5. Write the operation function

Create a new file in `lib/shopify/operations/` or add to an existing one. The function calls `storefront.request`, asserts the response is OK, applies caching, and returns the result:

```ts
import { cacheLife, cacheTag } from "next/cache";
import { defaultLocale, getCountryCode, getLanguageCode } from "@/lib/i18n";
import { assertStorefrontOk } from "../errors";
import { storefront } from "../storefront";

export async function getPrivacyPolicy(locale: string = defaultLocale) {
  "use cache";
  cacheLife("max");
  cacheTag("policies");

  const response = await storefront.request<ShopifyPrivacyPolicyResponse>(PRIVACY_POLICY_QUERY, {
    variables: {
      country: getCountryCode(locale),
      language: getLanguageCode(locale),
    },
  });
  assertStorefrontOk(response, "getPrivacyPolicy");
  const { data } = response;

  return data.shop.privacyPolicy;
}
```

Key points:

- **`storefront`** is the shared `@shopify/storefront-api-client` instance from `lib/shopify/storefront.ts`. Pass your response type as the generic; the query is the first argument and variables go in the options object.
- **`assertStorefrontOk(response, name)`** throws on a transport/GraphQL failure (no `data`), warns on partial errors, and narrows `response.data` to non-null.
- **`"use cache"`** uses the local Next.js cache for stable, low-cardinality reads
- **`cacheLife("max")`** caches until you manually invalidate
- **`cacheTag("policies")`** lets you invalidate with `revalidateTag("policies")` later
- Use `"use cache: remote"` instead for high-cardinality reads that depend on search, filters, sort, or cursors
- Always pass `locale` and extract `country`/`language` from it

## 6. Add a transform (if needed)

If the Shopify response shape differs from how your components want to consume the data, add a transform in `lib/shopify/transforms/`. This keeps Shopify-specific types out of your UI:

```ts
// lib/shopify/transforms/policy.ts
import type { Policy } from "@/lib/types";

export function transformShopifyPolicy(shopifyPolicy: ShopifyPolicy): Policy {
  return {
    title: shopifyPolicy.title,
    body: shopifyPolicy.body,
    slug: shopifyPolicy.handle,
  };
}
```

Then call it in your operation before returning:

```ts
return data.shop.privacyPolicy ? transformShopifyPolicy(data.shop.privacyPolicy) : null;
```

For simple responses where the shape already matches your needs, you can skip the transform.

## 7. Use it in a page

Call the operation from a Server Component:

```tsx
import { getPrivacyPolicy } from "@/lib/shopify/operations/policies";
import { getLocale } from "@/lib/i18n";

export default async function PrivacyPolicyPage() {
  const locale = await getLocale();
  const policy = await getPrivacyPolicy(locale);

  if (!policy) return notFound();

  return (
    <article>
      <h1>{policy.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: policy.body }} />
    </article>
  );
}
```

## Writing a mutation

Mutations follow the same pattern but without caching directives. Tag the mutation string with `#graphql ... as const` just like a query. If the mutation modifies cart state, you must call `invalidateCartCache()` afterward:

```ts
import { invalidateCartCache } from "@/lib/cart/server";
import { assertStorefrontOk } from "../errors";
import { storefront } from "../storefront";

export async function updateCartNote(note: string, cartId: string) {
  const response = await storefront.request<{ cartNoteUpdate: { cart: ShopifyCart } }>(
    UPDATE_CART_NOTE_MUTATION,
    { variables: { cartId, note } },
  );
  assertStorefrontOk(response, "updateCartNote");

  invalidateCartCache();
  return transformShopifyCart(response.data.cartNoteUpdate.cart);
}
```

Expose mutations to the client through server actions in `components/cart/actions.ts` using `"use server"`.

## Checklist

- [ ] Query string starts with `#graphql` and the declaration ends with `as const`
- [ ] Query/mutation is **named** so codegen can parse it
- [ ] Document is static — dynamic parts passed as variables, not interpolated into the query string
- [ ] `pnpm --filter template codegen` passes (validates fields against the live schema); generated types are gitignored and regenerate at dev/build
- [ ] `@inContext` directive included if data is locale-sensitive
- [ ] `storefront.request<ResponseType>(QUERY, { variables })` followed by `assertStorefrontOk(response, name)`
- [ ] `"use cache"` or `"use cache: remote"`, `cacheLife`, and `cacheTag` set for read operations
- [ ] Transform added if the Shopify response shape needs mapping
- [ ] Components import domain types from `lib/types`, not Shopify response types
- [ ] Cart mutations call `invalidateCartCache()`
