# Storefront Performance

Treat performance as part of the route design. Preserve the template's data, cache, and streaming contracts while replacing presentation.

## Start with the route contract

1. Read the project's `AGENTS.md` and the target route, its view contract, and the operations it calls.
2. Read the local Next.js documentation under `node_modules/next/dist/docs/` for any API being changed. This template may use newer Cache Components and prefetch behavior than model knowledge.
3. Classify each dependency as:
   - static or cacheable shell content;
   - request-time shared content;
   - request-time personalized content;
   - client-only interaction state.
4. Identify the expected LCP element and the content that must be interactive immediately.
5. Preserve existing promises, cache directives, invalidation tags, auth gates, redirects, metadata, and `notFound()` behavior unless the task explicitly changes them.

Read `references/shared-shell.md` for every route. Then read only the route references that apply:

- Home and marketing routes: `references/home.md`
- Collection and search routes: `references/plp-search.md`
- Product routes: `references/pdp.md`
- Cart and authenticated account routes: `references/cart-account.md`

## Build the rendering path

1. Put stable headings, primary media, and other LCP candidates in the static shell when the data contract permits it.
2. Push request APIs and request-dependent promises to the smallest component that needs them. Place a granular Suspense boundary there.
3. Give every visible fallback the same outer geometry as its resolved content. Do not replace a route with an unrelated full-page loader when independent sections can stream.
4. Start independent server work together. Use `Promise.all` or create promises before awaiting them; do not add serial Shopify waterfalls.
5. Keep components server-rendered by default. Move state, effects, browser APIs, and event handlers into leaf client components and pass the smallest serializable props.
6. Use `next/image` with truthful `sizes`, reserved aspect ratio, and intentional loading priority. Preload only the actual LCP image.
7. Treat prefetching as a traffic-versus-latency decision. Keep the default unless a production test shows that a higher-intent link benefits from more data being prefetched.

## Audit before finishing

Run the static hotspot scan from the skill directory:

```bash
node scripts/audit-storefront.mjs <storefront-root>
```

Treat its output as review prompts, not performance measurements. Then:

1. Run the storefront's lint, typecheck, and production build commands.
2. Test a production build, because prefetch behavior differs from development.
3. Verify direct visits and client navigations for every changed route.
4. Inspect the browser console, failed requests, layout shifts, the LCP request, and interaction behavior.
5. Use the Next.js bundle analyzer when client dependencies or boundaries changed.
6. Use field data from Speed Insights after deployment. Do not claim a performance improvement from code inspection alone.

Report which rendering contracts were preserved, what was measured, and any remaining risk.
