title: Why Use This
description: Why we built Vercel Shop — an agent-first Shopify starter for Next.js on Vercel.
type: overview
---

Most storefronts run on a full-stack commerce platform. The platform owns the theme, the rendering pipeline, the cart UI, and most of the surface customers see. It works until what you want to render exceeds what the theme supports, or the features you want to ship live on someone else's roadmap. Headless commerce moves the storefront onto your own stack. Your framework, your infrastructure, your render path. The commerce platform keeps doing commerce. Inventory, checkout, payments, tax.

Going headless means more of the stack is yours. Routing, caching, image optimization, and the rest of what a platform used to handle quietly. Templates and starters have tried to close that distance for a decade, and they keep failing in two ways: undernurtured (the README hasn't been touched since the framework had a different name), or so opinionated that customizing them costs you most of what you saved.

Meanwhile, the way teams write code has changed. Agentic development tools like Claude Code and Cursor keep getting meaningfully better at scaffolding across files, holding a framework's context, and producing software instead of suggestions. A starter aimed at agents isn't the same kind of artifact as a starter aimed at humans. It can be more opinionated and more flexible at the same time, because the opinions are documented in skills the agent reads, not enforced by abstractions you have to live with.

That's why we built Vercel Shop. It's a Shopify storefront in Next.js, deployed on [Vercel](https://vercel.com), with a set of skills that teach your agent how the codebase is shaped, how the Storefront API behaves, and how to keep changes deployable. Drop in your Shopify credentials and go. Swap in your CMS. Swap in your search. Wire up whatever you're convinced matters. The parts we kept stable are the boring ones on purpose.

The rendering model is Next.js 16's Cache Components: every route bakes its cacheable Shopify data into a `"use cache"` shell with a `cacheLife` window, and streams request-bound work — a selected variant, a filtered grid — into the smallest `Suspense` boundary that needs it. A Shopify webhook calls `revalidateTag()` on the affected cache tags, so an Admin edit surfaces immediately instead of waiting out the cache window. And Shopify integration isn't a bespoke client — `lib/shopify/storefront.ts` and customer authentication both build on `@shopify/hydrogen`, the same primitives Shopify ships and documents, so the template's conventions are Shopify's conventions.

Shopify has been chasing the same problem from its own end of the wire. Hydrogen used to look like a Shopify-flavored framework, its own opinions about routing and rendering bundled in alongside the commerce logic, which is a reasonable thing to build and also a heavy thing to ask a merchant to marry for a decade. The newer version backs off that ambition and narrows to what Shopify is actually best positioned to own: the storefront client, the cart, the customer session, built to run inside whichever rendering framework the rest of the industry is using instead of insisting on its own. Vercel Shop takes that at face value: it runs Hydrogen's own client and session helpers inside Next.js's cache and rendering model directly, rather than wrapping them in a translation layer that would just be new inessential complexity with a nicer name.

None of this makes the essential complexity of running a store disappear. Deciding what a product page should say is still work worth keeping for a human. A coding agent that can read a documented convention, follow a cache-tag policy, and check its own work against a real build is a genuinely new kind of collaborator for the other kind of complexity: the wiring, the boilerplate, the parts that were never about your store in the first place. Vercel Shop documents that wiring directly in skills an agent reads, so it gets productive on this codebase fast, instead of spending months guessing at conventions nobody explained to it.

A theme is something you use. A storefront built this way is something you own: a repository that answers to you, open to being changed because you had an idea, without a ticket filed with a vendor or a marketplace app catching up on its own schedule. Clone it, read it, and shape it into what your store actually needs.
